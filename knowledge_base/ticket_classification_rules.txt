===========================================
TICKET CLASSIFICATION RULES
Priority, Urgency, Routing & SLA Guidelines
===========================================

SECTION 1: PRIORITY DEFINITIONS
-------------------------------------------

P1 - CRITICAL
Business Impact:
- Complete service outage affecting multiple users
- Data breach or security incident in progress
- Revenue-generating system down
- Customer-facing application unavailable
- Data loss or corruption
- Legal or regulatory violation

Examples:
- Production database down
- Website completely inaccessible
- Active ransomware attack
- Payment processing system failure
- Email system down organization-wide

Response Time: 15 minutes
Resolution Target: 4 hours
Escalation: Immediate to on-call engineers
After Hours: Page on-call team
Updates: Every 30 minutes until resolved

P2 - HIGH
Business Impact:
- Significant degradation of service
- Security vulnerability identified
- Multiple users affected
- Important functionality unavailable
- Workaround available but inadequate
- Potential for escalation to P1

Examples:
- Application slow but functional
- Single server in cluster failed
- VPN intermittent connectivity
- Report generation failing
- Phishing email received
- Backup job failed

Response Time: 30 minutes
Resolution Target: 8 hours
Escalation: After 4 hours to senior engineer
After Hours: Email/SMS to on-call (no page)
Updates: Every 2 hours

P3 - MEDIUM
Business Impact:
- Isolated user issue
- Minor functionality problem
- Good workaround available
- Non-urgent feature request
- Standard service request
- Scheduled maintenance planning

Examples:
- Single user cannot print
- Minor UI bug
- Password reset needed
- Software installation request
- Access request for non-critical system
- Training request

Response Time: 2 hours
Resolution Target: 2 business days
Escalation: After 1 day to team lead
After Hours: Wait until next business day
Updates: Daily

P4 - LOW
Business Impact:
- Informational request
- Enhancement suggestion
- Documentation request
- Question or "how-to"
- Non-urgent cosmetic issue
- Future planning

Examples:
- Feature enhancement idea
- Documentation improvement
- General question about functionality
- Scheduled future changes
- Routine system maintenance

Response Time: 1 business day
Resolution Target: 5 business days
Escalation: After 3 days if needed
After Hours: Wait until next business day
Updates: When status changes

SECTION 2: URGENCY LEVELS
-------------------------------------------

CRITICAL URGENCY
Characteristics:
- Business operations stopped or severely impacted
- Immediate financial loss occurring
- Safety or security risk
- Legal/regulatory deadline at risk
- Executive-level visibility

Actions:
- Drop all other work
- Engage incident commander
- Mobilize response team
- Continuous communication
- Document all actions in real-time

HIGH URGENCY
Characteristics:
- Deadline within 24 hours
- Multiple users impacted
- Customer escalation
- Manager involvement
- Business process blocked

Actions:
- Prioritize over normal workload
- Assign senior resource
- Regular progress updates
- Identify workarounds
- Escalate proactively if at risk

MEDIUM URGENCY
Characteristics:
- Standard SLA applies
- Single user or small group affected
- Reasonable timeframe for resolution
- Minimal business impact
- Workaround available

Actions:
- Work in normal queue order
- Follow standard procedures
- Provide regular updates per SLA
- Document resolution

LOW URGENCY
Characteristics:
- No specific deadline
- Nice to have, not need to have
- No business impact if delayed
- Can be scheduled for convenience

Actions:
- Work during low-demand periods
- Batch with similar requests
- Optimize for efficiency
- May be deferred if higher priorities arise

SECTION 4: CATEGORY CLASSIFICATION
-------------------------------------------

INCIDENT vs REQUEST vs PROBLEM

INCIDENT:
- Unplanned interruption or reduction in service quality
- Something broke or isn't working as expected
- Requires restoration of normal service
- Focus on speed of resolution

Examples:
- Application crashed
- Server down
- Network outage
- Data corruption
- Performance degradation

REQUEST:
- Service Request (user asking for something)
- Planned and expected
- No service disruption
- Standard procedure exists

Examples:
- Password reset
- Software installation
- Access request
- New equipment order
- Information request

PROBLEM:
- Root cause of one or multiple incidents
- Requires investigation
- Long-term solution needed
- Prevents future incidents

Examples:
- Recurring application crashes
- Frequent network timeouts
- Pattern of account lockouts
- Systemic performance issues

SECTION 6: AUTOMATIC CLASSIFICATION RULES
-------------------------------------------

RULE 1: Security Keyword Detection
IF ticket contains: [hack, breach, ransomware, malware, phishing, suspicious, unauthorized, security, virus]
THEN:
- Category: Security
- Team: Security Team
- Priority: Minimum P2 (escalate to P1 if confirmed)
- Notify: Security operations center immediately

RULE 2: Production System Impact
IF ticket contains: [production down, prod error, customer-facing, revenue impact]
THEN:
- Priority: P1
- Urgency: Critical
- Team: Based on system (application/infrastructure)
- Notify: On-call engineer + management

RULE 3: Multiple User Impact
IF ticket contains: [all users, everyone, entire department, company-wide, nobody can]
THEN:
- Priority: Minimum P2
- Urgency: High
- Check: Potentially escalate to P1 if confirmed widespread

RULE 4: VIP User
IF ticket from: [executive, C-level, board member, VIP tag]
THEN:
- Urgency: Escalate one level
- Notify: IT management
- Assign: Senior engineer

RULE 5: Deadline Pressure
IF ticket contains: [urgent, ASAP, immediately, deadline, emergency, critical]
THEN:
- Review: Validate actual urgency with user
- Adjust: Priority based on business impact, not emotion
- Document: True business justification

RULE 6: Password Reset
IF ticket contains: [password reset, forgot password, locked out, cannot login] 
AND NOT contains: [multiple users, application]
THEN:
- Category: Service Request > User Account
- Priority: P3
- Team: End User Support
- Auto-Resolve: If using self-service portal

RULE 7: Access Request
IF ticket contains: [new user, new hire, onboarding, needs access, add user]
THEN:
- Category: Service Request > User Account
- Priority: P3
- Team: Identity & Access Management
- Require: Manager approval

RULE 8: Termination Request
IF ticket contains: [termination, last day, disable account, offboarding, resignation]
THEN:
- Priority: P1 (if today/immediate), P2 (if future dated)
- Team: Identity & Access Management
- Category: Service Request > User Account
- Notify: Security team
- Action: Disable access by specified date/time

SECTION 8: ESCALATION PROCEDURES
-------------------------------------------

TIER 1: First Level Support
- Initial ticket triage and classification
- Basic troubleshooting (password resets, restarts)
- Known issues with documented solutions
- Information gathering
- User communication
- Escalate after: 30 minutes for P1, 2 hours for P2

TIER 2: Technical Specialists
- Advanced troubleshooting
- System configuration changes
- Root cause analysis
- Coordination with other teams
- Escalate after: 2 hours for P1, 4 hours for P2

TIER 3: Subject Matter Experts
- Complex technical issues
- Architecture decisions
- Vendor engagement
- Deep system knowledge
- Escalate after: 4 hours for P1

MANAGEMENT ESCALATION:
- P1 not resolved in 4 hours
- P2 not resolved in 12 hours
- Customer escalation or complaint
- Policy exception needed
- Resource contention issues

SECTION 9: QUALITY CHECKS
-------------------------------------------

BEFORE CLOSING TICKET:
□ Issue completely resolved (not just worked around)
□ Root cause identified and documented
□ User confirmed resolution
□ Knowledge base article created/updated if new issue
□ Related tickets linked
□ Time logged accurately
□ Classification correct (category, priority, team)
□ All required fields completed
□ Follow-up scheduled if needed

CUSTOMER SATISFACTION SURVEY:
- Sent automatically upon ticket closure
- Tracked metrics: Resolution time, communication, quality
- Poor ratings trigger manager review
- Trends analyzed monthly

TICKET CLASSIFICATION ACCURACY:
- Random sample audited weekly
- Feedback provided to analysts
- Retraining if accuracy < 90%
- AI classification training data
